Yes — an AI agent can estimate a tree’s height and trunk diameter (DBH) pretty well from photos/video, and then estimate weight/biomass from forestry formulas. But it can’t directly “see weight” — weight is always an estimate derived from measurements + species assumptions.

Here are the best approaches (from most accurate to easiest to ship):

1) Best accuracy: Phone LiDAR (iPhone Pro/iPad Pro) + AI

If the user has a LiDAR device, your app can:

build a quick 3D point cloud,

measure height and trunk diameter,

estimate crown volume.

This is the closest thing to “push a button, get measurements.”

2) Solid and scalable: Photogrammetry (video walk-around)

User records a short video circling the tree (10–20 seconds). Your backend:

reconstructs a 3D model (structure-from-motion),

measures height/diameter,

flags low-confidence scans.

Works on most modern phones, but needs good lighting and enough texture.

3) Fastest UX: Single photo + reference object

Have the user include a known reference:

a person (with entered height),

a measuring stick/pole,

an 8.5×11 sheet on the trunk,

or even a credit card for bark-scale closeup.

AI does:

detects tree + trunk,

estimates DBH from the closeup,

estimates height from full-tree photo using the reference,

outputs a confidence score.

This is the easiest to implement reliably without special hardware.

4) “Forestry standard” method: Clinometer-style using AR

Your app guides the user to:

stand a known distance away (AR measures distance),

aim at top and base,

compute height with basic trig.

Then DBH is measured by:

AR “wrap” estimate around trunk, or

one closeup photo with a reference card.

Very accurate when users follow instructions.

Estimating weight (what you can claim)

Once you have:

DBH (diameter at breast height)

Height

Species (user selects or AI guesses with confirmation)

You estimate:

Green weight (fresh-cut weight)

Dry weight

Biomass

Optionally log weight by section length

This uses established forestry “allometric” equations. But species matters a lot, so best practice is:

AI suggests top 3 species → user confirms.

What you should build in your app (recommended workflow)

Pick mode

“Quick Estimate (1–2 photos)”

“Accurate Scan (video / LiDAR)”

Capture guidance

show silhouette overlay

require: base visible + top visible

require: reference object or entered reference height

Outputs

Height (ft)

DBH (in)

Estimated total weight range (lbs)

Confidence (Low/Med/High)

Notes: “requires crane likely” / “access issues detected” (optional)

Insurance/estimating tie-in

Convert to scope inputs: DBH class, height class, hazard/access class, haul distance

Generate apples-to-apples comparisons from those standardized classes

Accuracy expectations (be honest in the UI)

Height from LiDAR/video: often within ±3–8% (good conditions)

Height from single photo: ±8–20% depending on reference + angle

DBH from closeup + reference: can be very good (±0.5–1.5 in) if aligned

Weight: usually a range (often ±15–35%) because species, moisture, hollows, and branching vary
